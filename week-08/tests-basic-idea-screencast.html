Basic idea

  • Tests make sure your functions always return the expected result in well-defined case
  • Start with a few obvious cases, corner solutions, wrong input that may conceivably happen
  • Bugs always become test cases — assures you do not re-introduce errors
  • Always run your tests after changing your code

Why bother about testing?

  • We make errors. Period.
  • Usual mode: Continue so long as output looks plausible.
  • Means bugs will crop up later when you have forgotten about the code.
  • Buggy code is biased towards expected results
  • The more you invest in quality, the less total time it takes to get your code to work

Testing the right thing: Interfaces vs. implementation

  • Difference is one of the most important ideas in computing

    Interface How something interacts with the world

    Implementation How it does what it does

Testing the right thing: Interfaces vs. implementation

  • Simple example:

    def integrate(func, x1, x2):
    ... math goes here ...
    return result

    Interface {func, x1, x2}→ integral

    Implementation We don't (have to) care

Design for test

  • Design components so they rely exclusively on interfaces
  • Empirics show that interfaces are much more stable
  • Isolate interactions with outside world (e.g., opening files)
  • Make things you are going to examine deterministic
  • In short: write pure functions
