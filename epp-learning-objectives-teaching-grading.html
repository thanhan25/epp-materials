Learning objectives

Modern didactics starts by defining a set of active knowledge you should have developed at the end of the course. The course contents are then derived from that and so is the teaching style. Finally, the things we require you to do in order to pass the course are designed to require exactly those skills that we hope you will develop.

We group the learning objectives into four categories: - Methods (core concepts and abstract coding skills) - Tools (specific programmes, which you may want to use in the long run or not) - Personal skills - Social skills

We will now describe each of these in turn.

Methods

  • Students master the basic techniques to make current and past versions of their research reproducible
  • Students solve reasonably complex economic problems using the help of a computer
  • Students are able to write correct code in an efficient manner employing modern software engineering techniques
  • Students reason about the structure of economic data and handle such data efficiently

Tools

  • Students use a shell for navigating the directory system and for starting programmes
  • Students routinely use Git for version control and collaborating with coauthors
  • Students manage economic data using basic pandas functionality
  • Students automate repetitive tasks using Python
  • Students visualize data using seaborn and matplotlib
  • Students leverage build frameworks for running analysis pipelines (e.g., pytask, waf)
  • Students speed up their code using numpy and numba
  • Students write automated unit tests using the pytest framework
  • Students find solutions to specific programming problems through sensible use of search engines and websites like Stack Overflow

Personal

  • Students manage their time efficiently when solving long and complex tasks
  • Students seek for help when stuck and explain their problems in a comprehensible fashion
  • Students increase their frustration tolerance when pushing the limits of their coding abilities
  • Students reflect on lecture material and distill the important gists themselves
  • Students employ the scientific method when improving the correctness and efficiency code
  • Students reflect on their own contribution to collaborative work

Social

  • Students collaborate effectively for solving tasks of varying complexity
  • Students reason about code and solution strategies in groups and in plenary sessions
  • Students explain the contents and takeaways of preparation materials and in-class activities to group members who find it hard to follow along
  • Students assess different dimensions of contributions to constructive group work

The flipped classroom concept

Between 2010 and 2019, I taught this course in a somewhat traditional manner. I lectured on programmes, coding styles, best practices, etc.. Students were required to show they had a grasp on these by means of challenging homeworks and a final project. While student feedback had generally been good, I was unable to overcome two challenges using this concept. Essentially, there was so much heterogeneity in prior knowledge (note that when it comes to the contents of this course, there is nothing like a basic course in macroeconomics or econometrics to build upon), that it was impossible to pace the lecture. Typically, one third of the course found it way too fast, another third way too slow, and the remaining third just about right. While there was no way to closely observe the nature of group work for the assignments, we always had the impression that in the end of the day, one or two people carried groups and others were dragged along, leading to frustration and little learning.

Having had good experiences with the flipped classroom concept in the Summer 2020, I will now employ it in this course as well. It allows to develop all dimensions of the learning objectives in a meaningful fashion.

The weekly schedule will be as follows:

  • Before class, there will be:
    • Self-paced learning of concepts / knowledge. We will provide you with screencasts, literature, or short programming problems that will allow you to acquaint yourself at your own pace with the material.
    • Short quizzes, which allow you to check your knowledge. Note that these do not constitute tests. They do not enter the grading in any way, they are for you to check whether you understood a particular concept.
  • During class, there will be
    • Carefully designed group work in order to deepen your understanding of certain topics
    • Class discussions to collect the results from the group work and to foster presentation skills
    • Evaluation of everybody’s contributions to group work via the Pentabilities system

At a lower frequency, there will be: - Challenging homeworks, which you will partly work on during class time - Personalised feedback from the Pentabilities system and your reflection on it. - A final project that you should be able to use in another context, too.

Combined, these elements should allow you to - acquire the basic knowledge at your own pace - deepen it in group work - improve your personal skills in self-directed learning - improve your social skills with the help of the Pentabilities system

Assessment

As the teaching style, our choice of assessment criteria is guided by the learning objectives. That is, the different components require your skills along the dimensions of methods, tools, personal skills, and social skills. We need to respect the rules set by the examination regulations; otherwise the weighting of the components would be much more uniform.

In particular, there will be 1. plenty of group work on challenging projects (relevant for all four dimensions), 1. accompanied by Pentabilities (personal and social skills), 1. and a final project of your choice (methods, personal skills, and tools unless you want to use other tools, which is perfectly fine), which would ideally be useful for you in other contexts, too.

The remainder of this section is about the precise rules for grading and for the final project. Please also see the document on Pentabilities for that component. Note that large parts of the section on the final project will only become comprehensible as the course progresses.

Rules for the Final Project

Overall, you should convince us that you have a met the learning objectives of this course that you find under “methods”. I.e., very abstractly, you are able to solve interesting economic problems that require the help of a computer in a way that is reproducible by others.

We will agree on a topic for each final project during the last session, i.e., on 11 February 2021. You should then provide us with a set of script(s) that you wrote to solve a particular problem on 14 March 2021 at the latest. This could be, for example, polished code from a seminar paper that you wrote; or some scripts to automate annoying, tedious tasks.

Here are some more detailed notes. Most probably, they are not comprehensible before the course but they will be helpful once you get started.

  1. Please work on Github.

    The last time I checked, GitHub placed a hard limit of 100MB on individual files. So do not include such files in your repository. If you are a PhD student, you want to keep larger data under VC, and you will continue working on the project, please ping me for an alternative service.

  2. Grading will be based on programming style, task complexity, how much time it takes to understand where you are headed, whether your results are reproducible (depends a bit on the project), and your use of Git.

    Making use of the project templates is by no means a requirement, but doing so will be useful for most projects. Webscraping is the clear exception here (i.e., this would be a separate step before applying the pytask machinery to the scraped data). In any case, it needs to be clear why the structure you chose for your project is useful for what you want to get done.

  3. It is no requirement that we are able to run your code. However, if no data or software requirements restrictions apply, doing so makes things easier for us. But you should use whatever data or software are most useful for you.

  4. In any case, provide a README that specifies the environment needed to run your project. That is, anything not included in your Github repository. This includes all required software (including things like Python libraries, R packages, Stata ado-commands, …), on which operating system it has been tested, external data, etc.. If you just use Python and/or R, the most helpful strategy is to use a conda environment file.

  5. When you are done, place a tag called epp-final-project on the commit that we should look at. Add me (hmgaudecker), Gregor Böhl (gboehl), and Janos Gabler (janosg) to the project.
