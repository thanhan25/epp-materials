Data Management — General rules and Normal Forms

Hans-Martin von Gaudecker

Definition and general rules

  • Converting the source data to the format(s) your analysis programs (estimators, simulators, calibrators) expect.
  • Source data always refers to the original data — as downloaded, collected, or simulated.
  • Never, ever, change these source data and save it under the same name. The source data should always be the starting point of your workflow.
  • Similarly, never change the contents of a variable (and certainly not later on in the workflow — minimise state!).

The first normal form

  • Values do not have any internal structure
  • I.e., no need for parsing values before using them
  • E.g. store first names and last names separately
  • Not too often a problem in economic data
    • X-digit industrial or educational classifiers
    • Store each digit level you need in a separate variable

The second normal form

  • Tables do not contain redundant information
  • In a panel structure: Store time-constant characteristics in a separate table
  • Violations make things much harder and error-prone:
    • Changes to data
    • Consistency checks
    • Selecting observations

No structure in variable names

  • There should not be different variables with similar content referring to different time periods etc.
  • In our example: Could store one row per individual, append _2019_11, _2020_04 to variable name
  • In Stata parlance: Use “long” instead of “wide” format
